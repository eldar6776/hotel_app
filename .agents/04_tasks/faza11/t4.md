## 3. TEHNICKI ZAHTJEVI

### 3.1 Baza podataka — `work_orders` tabela
- `id` (UUID, PK)
- `room_id` (UUID, FK → rooms.id)
- `category` (ENUM: HVAC, PLUMBING, ELECTRIC, TV, FURNITURE, OTHER)
- `description` (TEXT, not null)
- `priority` (ENUM: LOW, MEDIUM, HIGH, EMERGENCY)
- `status` (ENUM: PENDING, IN_PROGRESS, COMPLETED, CANCELLED, default: PENDING)
- `reported_by` (UUID, FK → users.id) — ko je prijavio (sobarica, recepcioner, gost)
- `assigned_to` (UUID, FK → users.id, nullable) — tehničar
- `resolution_notes` (TEXT, nullable)
- `completed_at` (TIMESTAMP, nullable)
- `created_at`, `updated_at` (TIMESTAMP)

### 3.2 API endpointi
- `GET /api/work-orders` — lista sa filtriranjem (status, category, priority, room_id)
- `POST /api/work-orders` — kreiranje novog work ordera
- `PUT /api/work-orders/{id}` — update statusa, assignment, notes
- `GET /api/work-orders/stats` — statistika: broj po kategoriji, prioritetu, statusu, prosječno vrijeme rješavanja

### 3.3 Poslovna logika
- Automatska notifikacija tehničkom osoblju kada se kreira work order sa HIGH ili EMERGENCY prioritetom
- Historija popravki po sobi (listanje work orders za dati room_id)
- Prioritet EMERGENCY automatski eskalira odgovornom menadžmentu
