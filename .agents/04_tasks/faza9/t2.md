## 1. CILJ

## 3. TEHNICKI ZAHTJEVI

### 3.1 Entiteti
```
ProformaInvoice.cs
  - Id (Guid), BookingId (Guid)
  - ProformaNumber (string) — posebna sekvenca
  - IssueDate (DateTime)
  - TotalAmount (decimal)
  - Status: ProformaStatus (Issued, PartiallyPaid, FullyPaid, Cancelled)
  - ExpiryDate (DateTime?) — rok vazenja
  - ConvertedToInvoiceId (Guid?) — kada se pretvori u pravi racun

AdvancePayment.cs
  - Id (Guid), BookingId (Guid)
  - Amount (decimal)
  - PaymentDate (DateTime)
  - PaymentMethod (PaymentMethod)
  - Reference (string?)
  - AppliedToInvoiceId (Guid?) — racun na koji je primijenjeno
  - IsRefunded (bool)
```

### 3.2 API Endpointi
```
POST   /api/proforma                    — kreiranje predracuna
GET    /api/proforma/{bookingId}        — predracun za booking
POST   /api/proforma/{id}/convert       — pretvaranje u pravi racun
POST   /api/advance-payments            — evidentiranje avansne uplate
GET    /api/advance-payments/{bookingId} — lista avansnih uplata
POST   /api/advance-payments/{id}/refund — storno avansa
```

### 3.3 Poslovna logika
- Proforma se generise na zahtjev (ne automatski)
- Proforma ima rok vazenja (default: 7 dana)
- Avansna uplata se moze primijeniti na jedan ili vise racuna
- Pri konverziji proforma → invoice:
  - Kreirati pravi racun (T9.1)
  - Primijeniti avansne uplate na novi racun
  - Oznaciti proformu kao Converted

### 3.4 Validacija
- Proforma se ne moze kreirati za booking sa statusom CheckedOut (vec je racun generisan)
- Avans ne moze biti veci od totala
- Refund je moguc samo ako avans nije primijenjen na invoice
- Proforma broj ima format: `PRO-GODINA-BROJ`
