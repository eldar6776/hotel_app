## 3. TEHNICKI ZAHTJEVI

### 3.1 Availability check engine
```
IBookingAvailabilityService
  CheckAvailability(request: AvailabilityRequest): AvailabilityResult
  LockRooms(request: LockRequest): LockResult
  ReleaseRoomLock(bookingId: Guid, roomTypeId: Guid): void

AvailabilityRequest
  - RoomTypeId (Guid)
  - Arrival, Departure (DateTime)
  - Quantity (int) — broj soba
  - ExcludeBookingId (Guid?) — za izuzimanje pri updateu

AvailabilityResult
  - IsAvailable (bool)
  - AvailableQuantity (int)
  - ConflictingPeriods (List<DateRange>)
```

### 3.2 Double-booking prevention
- Query: prebrojati sve BookingRoom u periodu [Arrival, Departure-1] gdje je status != Cancelled i Status != Released
- Ako `sum(quantity) + requested > total_rooms_of_type` → nije dostupno
- Overbooking dozvoljen samo ako je konfigurisan parametar `AllowOverbooking` = true

### 3.3 Real-time locking
- Koristiti `SELECT ... FOR UPDATE` (PostgreSQL row-level locking) na BookingRoom tabeli
- Lock se dobija prije provjere i drzi do potvrde rezervacije
- Timeout: 5 sekundi, ako se ne dobije lock → vratiti 409 Conflict
- Transaction scope: cijeli booking flow mora biti u jednoj transakciji

### 3.4 API Endpointi
```
GET  /api/availability?roomTypeId={id}&arrival={date}&departure={date}&quantity={n}
     — provjera dostupnosti (read-only, bez lockinga)
POST /api/availability/check-and-lock
     — provjera + lock za konkretan booking flow
```
