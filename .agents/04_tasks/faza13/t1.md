## 3. TEHNICKI ZAHTJEVI

### 3.1 Booking.com API konekcija
- Autentifikacija: API key + autentifikacija putem Booking.com Connectivity API
- Rate limiting: poštovati Booking.com limit (100 req/min default)
- Endpointi:
  - `GET /api/channels/booking/reservations` — preuzimanje novih rezervacija
  - `POST /api/channels/booking/reservations/{id}/confirm` — potvrda rezervacije
  - `POST /api/channels/booking/reservations/{id}/cancel` — otkazivanje
  - `GET /api/channels/booking/availability` — stanje dostupnosti

### 3.2 Webhook handler (primanje notifikacija)
- Webhook URL: `POST /api/webhooks/booking`
- Obrada eventa: `new_reservation`, `cancellation`, `modification`
- Validacija webhook potpisa (Booking.com signature header)
- Kreiranje interne rezervacije na osnovu webhook podataka

### 3.3 Mapiranje podataka
- Booking.com room ID → HotelPRO room_type_id
- Booking.com rate plan → HotelPRO rate_plan_id
- Gost podaci: ime, email, telefon, specijalne napomene

### 3.4 Logging i monitoring
- Log svih API poziva i webhook primitaka u `channel_logs` tabelu
- Dashboard za status konekcije (connected/disconnected/error)
- Alert ako nije bilo webhooka u zadnjih 60 minuta

## 5. LEGACY MAPPING

### 5.1 Legacy proces → Novi API

| Legacy proces | Opis | Novi endpoint | Napomena |
|---|---|---|---|
| Ručni unos Booking.com rezervacija | Recepcionar unosi ručno | `POST /api/webhooks/booking` | Automatski webhook |
| XML channel sync | Stari XML format | JSON + REST API | Modernizacija formata |
| Ručna sinhronizacija cijena | Excel upload | `POST /api/channels/booking/rates` | Automatski sync |

### 5.2 Legacy logika → Nova logika

- Legacy: Ručni unos svake rezervacije → Novo: Webhook automatski kreira booking
- Legacy: Cijene se ažuriraju ručno na portalu → Novo: Dvosmjerna sinhronizacija
- Legacy: Nema rate limiting → Novo: Poštovanje Booking.com limita (100 req/min)

## 6. RESTRIKCIJE
- Webhook signature se obavezno validira — neprihvaćeni zahtjevi se odbijaju sa 401
- Ne slati booking podatke u logove (GDPR — maskirati lične podatke)
- Rate limit se mora poštovati — implementirati queue za emitovanje promjena
- Ne dozvoliti duple rezervacije (provjera po `booking.reservation_id`)
