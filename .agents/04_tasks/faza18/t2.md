## 3. TEHNICKI ZAHTJEVI

### 3.1 Performance baseline
- Izmjeriti trenutne performanse kljucnih endpointa:
  - `/api/bookings` (lista + filtriranje)
  - `/api/availability/search` (pretraga kapaciteta)
  - `/api/folio/{id}/transactions` (folio transakcije)
  - `/api/dashboard/summary` (dashboard sa agregacijama)
- Alati: `k6` (Grafana k6) za load testing
- Baseline se cuva u `performance/BASELINE.md`

### 3.2 Load test scenariji
- **Daily peak**: 50 concurrent users, 100 req/s, 15 min
- **Weekend peak**: 100 concurrent users, 200 req/s, 30 min
- **Season peak**: 300 concurrent users, 500 req/s, 60 min
- **Stres test**: postepeno povecanje do 500 concurrent users
- Kriterijumi: p95 response < 500ms, error rate < 1%

### 3.3 Query optimizacija
- Identifikovati N+1 query probleme (EF Core Include/ThenInclude)
- Dodati indekse na kolone koje se cesto filtriraju (Status, Date, GuestId, PropertyId)
- Optimizovati booking availability query (trenutno najzahtjevniji)
- Koristiti `AsNoTracking()` za read-only queryje
- Razmotriti `Table Per Hierarchy` vs `Table Per Type` za polimorfne entitete

### 3.4 Caching strategija
- **Redis cache** za:
  - Room availability (TTL: 5 minuta)
  - Dashboard agregacije (TTL: 10 minuta)
  - Konfiguracija i setup (TTL: 60 minuta)
- **Response caching** za statičke podatke (room types, rate plans, services)
- **In-memory cache** za sesije i JWT revocation

### 3.5 Frontend optimizacija
- Lazy loading za sve rute osim pocetne
- Image optimizacija (next/image)
- Bundle size analiza (webpack-bundle-analyzer)
- Code splitting za velike komponente (kalendar, izvjestaji)
